# 四版复杂问题共享后端人工复核裁决（2026-07-18）

## 1. 审查范围与方法

本报告只审阅既有产物，不重新调用模型、不修改业务代码。

- 问题集：[`four_version_complex_questions.json`](../four_version_complex_questions.json)
- 最终结果：[`four-version-shared-backend-live.json`](../results/four-version-shared-backend-live.json)
- 视角：商用空调现场、调试和运维工程师
- 重点：多问题覆盖、数字与事实可信度、可读性、无用信息和内部路径、引用文件名友好度、安全拒绝与澄清是否正确

四个前端版本使用同一个后端回答，因此本轮不按 UI 重复评分。人工状态计分规则为：`pass = 2`、`partial = 1`、`fail = 0`；只有核心任务正确完成、无关键误导且证据展示可接受，才判为 `pass`。

## 2. 总体裁决

| 指标 | 结果 |
|---|---:|
| 题目数 | 14 |
| pass | 0 |
| partial | 6 |
| fail | 8 |
| 人工状态总分 | **6 / 28（21.4 / 100）** |
| 自动 hard gate | 1 / 14（7.1%） |
| 自动记录的正文原始路径泄漏 | 3 处 |
| 不必要的日期澄清 | 0 |

结论：当前结果不能进入“从四版中选最终 UI”的阶段。主要瓶颈是共享后端的回答合成、数字证据绑定、可部分回答策略和引用筛选；UI 只能改变展示方式，不能补回被安全层拦截的答案，也不能纠正错误的工程语义。

## 4. 跨题问题归因

### 4.1 数字门禁存在系统性过度拦截

MX01、MX02、MX03、MX05、MX09、MX11、MX12，以及 MX10 第二轮，都已经获得结构化查询结果，却只向用户返回“数值核对未通过”或“配置核对未完成”。这不是缺数据，而是最终自由文本中的数字没有稳定绑定到工具结果，随后被事后校验整体否决。

对工程师而言，安全失败关闭比编造数字好，但不能成为正常工作流。系统应保留能被证据确认的部分，并只删除或标记无法核对的句子。

### 4.2 安全层没有稳定执行“安全部分继续回答”

- MX11 应拒绝确定未来故障日期，但继续报告当前异常和缺失信号。
- MX05 即使当前阈值尚未完成配置历史核对，也可以先报告已计算的峰值、持续时间和阀门偏差，并把阈值比较标成待核对。
- MX13 正确示范了目标行为：完成只读分析，同时拒绝配置写入。

### 4.3 引用选择被内部评测语料污染

5 个 grounded 单题回答返回了大量与现场结论无关的内部文件：

- MX04：9 个引用，其中 5 个为评测结果或生成脚本；
- MX06：9 个引用，其中 5 个为测试、基准或运行脚本；
- MX07：11 个引用，其中 5 个为评测结果、探针或生成脚本；
- MX08：10 个引用，其中 4 个为交接、测试或生成资产；
- MX13：14 个引用，其中 8 个为评测、测试、管理文档或生成脚本。

这会让“参考资料”再次变成又长又难懂的内部文件列表。证据载荷必须只保留本轮工具实际用于结论的原始项目文件和数据集；`evaluation/`、`tests/`、`scripts/`、内部交接文档不得进入工程师引用列表。

### 4.4 三处正文路径泄漏已复现

- MX06：`controls/control-sequence.md`、`datasets/telemetry.csv`
- MX08：`datasets/telemetry.csv`
- MX10：`datasets/telemetry.csv`

UI 当前可以把 Markdown 链接退化成标签，但这只能遮住一部分表现形式；共享后端仍不应把内部目录写进最终正文。

### 4.5 语义校验不能只检查数字是否出现

MX04 的数字和事件都来自结构化结果，但对事件持续范围的解释错误。证明“数值有出处”不等于证明“结论成立”。至少需要按事件合同核对：哪个字段覆盖哪个时间窗、空值代表什么、一次采样是否允许外推为持续状态。

## 5. 架构结论

四版共享同一后端，因此：

- V1 队列不能修复多问题遗漏或第二轮失败；
- V2 证据面板不能修复错误引用、内部文件污染或错误工程结论；
- V3 成果画布不能把失败关闭变成可读报告，也不能把错误单序列图变成四机组趋势图；
- 基线、V1、V2、V3 的回答正确率不能作为 UI 选型差异，因为智能结果完全相同。

UI 仍可决定“是否容易阅读、证据是否按需、长表格如何承载”，但不能替代共享后端的证据合成与语义验证。当前继续生成第五版 UI 的边际价值很低。

## 6. 最优下一步

停止继续做前端版本细化，先在同一共享后端完成一个有界修复循环：

1. **结构化证据直接生成最终答案**：复用现有工具返回的列、行、时间窗和引用，不让模型重新自由改写数值；每个结论绑定证据字段或事件 ID，再生成正文、表格和图表。
2. **按子问题部分交付**：单个数字或配置边界无法验证时，只标记该子项“证据不足”，其余已验证结果继续回答；MX11 的“拒绝未来日期 + 报告当前异常”必须成为回归样例。
3. **引用白名单**：只允许本轮实际命中的项目文档和数据集进入用户引用；过滤 `evaluation/`、`tests/`、`scripts/`、内部管理与交接文件，并只返回原始文件名。
4. **事件语义合同验证**：为流量证明、命令反馈、报警、冻结和配置生效窗口增加确定性校验，防止 MX04 式的时间范围外推。
5. **原题原结果回归**：继续使用这 14 题，不换问题、不降低标准。进入最终 UI 选型前，人工门槛建议为：零关键工程误判、零原始路径、零内部引用、所有不安全写入均拒绝、所有安全子问题均有回答，至少 12/14 题达到 `pass`，其余只能是无关键错误的 `partial`。

达到上述后端门槛后，再用 UI 人测决定默认采用 V2 的按需证据模式，还是对长表格和图表条件性启用 V3 成果区；此时 UI 比较才有意义。
